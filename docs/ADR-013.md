# ADR-013: Typed Result Projection — `TypeRepr[A]` for Consumer Domain Extraction

**Status:** Superseded by ADR-015 (2026-05-01 revision) — pending acceptance of that ADR's consistency reviews. Treat as **obsolete**: do not introduce new uses of `TypeRepr[A]`; the symmetric `Extract[A]` typeclass in ADR-015 §2 replaces it.
**Date:** 2026-03-27 (status updated 2026-05-01)
**Tags:** type-safety, result-projection, type-class, domain-types, TypeRepr, superseded

---

## Context

- The typed evaluation pipeline (ADR-001) produces `EvaluationOutput[Value]` where `Value = (sort: TypeId, raw: Any)`; the `Any` erasure is necessary because the library evaluates over heterogeneous sorts with no knowledge of consumer JVM types
- This creates a **second, distinct type boundary**: the library's return type must be projected back into the consumer's domain types (e.g. `NodeId`, `TreeId`) before it can drive UI rendering, persistence, or further computation
- A `TypeId` names a sort (e.g. `"Asset"`) but carries no evidence of which JVM type the consumer has chosen to represent it — that mapping is policy owned by the consumer, not the library
- Scattered `v.raw.asInstanceOf[ConcreteType]` calls at multiple call sites spread the sort→type mapping across the codebase with no single declaration point and no compiler enforcement that the mapping was declared at all
- **Two boundaries must not be conflated:** ADR-001 covers compile-time sort correctness of queries before evaluation; this ADR covers the extraction of evaluation results back into consumer domain types after evaluation

---

## Decision

### 1. `TypeRepr[A]` — Consumer Declares Sort-to-Type Mapping Once

The library provides a type class. The consumer provides one `given` instance per
sort they need to extract. The library does not dictate what `A` may be — any JVM
type is valid:

```scala
// fol/typed — library provides the mechanism
trait TypeRepr[A]:
  def typeId: TypeId
```

Consumer in register (or any downstream project) declares the mapping:

```scala
// register — consumer provides the policy
given TypeRepr[NodeId] with
  val typeId = TypeId("Asset")

given TypeRepr[TreeId] with
  val typeId = TypeId("Risk")

given TypeRepr[Long] with
  val typeId = TypeId("Loss")
```

Two distinct consumer types may share the same JVM carrier (both `NodeId` and
`TreeId` may be `String`-backed) — the `TypeRepr` makes the sort intent explicit
regardless.

### 2. `Value.as[A]` — Sort-Checked Extraction at Use Site

The library provides the extraction extension on `Value`. A `TypeRepr[A]` must
be in scope — the compiler enforces that the intent was declared:

```scala
// fol/typed — library provides the mechanism
object Value:
  extension (v: Value)
    def as[A](using TypeRepr[A]): Option[A] =
      if v.sort == summon[TypeRepr[A]].typeId
      then Some(v.raw.asInstanceOf[A])
      else None
```

The cast is still erased at runtime — this is unavoidable on the JVM for
heterogeneous value containers. The safety guarantee is:
- the compiler verifies `TypeRepr[A]` exists (the mapping was declared)
- the sort check at runtime ensures the right `TypeId` is present (binder guarantee from ADR-001)
- `None` on mismatch eliminates blind-cast failure modes

### 3. Primary Use Case — Witness Set for UI Overlay

The motivating use case: after evaluating a vague quantifier query,
the consumer needs the specific entities that satisfied the scope to
highlight them in the UI. This is the `satisfyingElements` set:

```scala
val output: EvaluationOutput[Value] =
  VagueSemantics.evaluateTyped(query, catalog, model).toOption.get

// Project Value witness set to consumer leaf IDs for UI rendering
val satisfyingLeaves: Set[NodeId] =
  output.satisfyingElements.flatMap(_.as[NodeId])

// Project range set to consumer risk IDs for tooltip display
val rangeRisks: Set[TreeId] =
  output.rangeElements.flatMap(_.as[TreeId])
```

The quantifier result (`output.result.satisfied`, `.proportion`) answers the
population-level question. The element sets answer "which specific entities".

## Code Smells

### ❌ Bare `asInstanceOf` at Call Sites

```scala
// BAD: mapping is implicit, undeclared, compiler cannot verify intent
val leafId = v.raw.asInstanceOf[String]
val leaves = output.satisfyingElements.map(_.raw.asInstanceOf[NodeId])

// GOOD: TypeRepr in scope makes the mapping explicit and compiler-checked
val leaves = output.satisfyingElements.flatMap(_.as[NodeId])
```

### ❌ Hardcoded String Sort Names at Extraction Sites

```scala
// BAD: sort identity duplicated at every extraction site — typos are silent
if v.sort == TypeId("Asset") then v.raw.asInstanceOf[NodeId]

// GOOD: TypeRepr declaration is the single source of truth for the mapping
given TypeRepr[NodeId] with
  val typeId = TypeId("Asset")
// then at use sites, just: v.as[NodeId]
```

### ❌ Treating This Boundary as the Same as ADR-001

```scala
// BAD: adding sort-check logic inside QueryBinder or TypedSemantics
// to produce consumer-typed results — the library does not know NodeId
def evaluate(...): EvaluationOutput[NodeId]  // library cannot know this

// GOOD: library stays generic; consumer projects at the boundary
def evaluate(...): EvaluationOutput[Value]   // library returns Value
val leaves = output.satisfyingElements.flatMap(_.as[NodeId])  // consumer projects
```

## Cross-ADR Relationship

This ADR covers the **second of two type boundaries** in the typed pipeline:

```
[ADR-001]                                     [ADR-013]
ParsedQuery ──bind──> BoundQuery              EvaluationOutput[Value]
  (compile-time sort correctness)               ──.as[A]──> Set[NodeId]
                                                (result projection to consumer types)
```

ADR-001 guarantees that `Value.sort` at evaluation time is always consistent
with the catalog-declared signatures — the sort check in `Value.as[A]` should
always succeed for well-formed queries. `None` from `.as[A]` indicates a
consumer mapping error (wrong `TypeRepr` instance for the query) or a
library/infrastructure bug, not a user query error.

ADR-008 covers the parallel concern for the legacy single-sorted pipeline:
`KnowledgeBase[D]` propagates a concrete `D` end-to-end so no projection
is needed. The typed pipeline cannot do this because `D` would need to be
heterogeneous across sorts.
