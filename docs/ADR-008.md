# ADR-008: Domain Type Safety — Generic `KnowledgeBase[D]`

**Status:** Accepted  
**Date:** 2026-03-25  
**Supersedes:** ADR-006 (Domain Type Erasure in KB-Backed Models)  
**Tags:** type-safety, generics, domain, type-classes

---

## Context

- ADR-006 accepted `Model[Any]` as a pragmatic shortcut for KB-backed models, deferring full type safety
- `RelationValueUtil.toDomainValue: RelationValue => Any` was the single erasure point, discarding the ADT sum type
- `NumericAugmenter.toDouble` performed runtime dispatch on `Any` without compiler exhaustiveness checking
- The downstream register project needs a richer domain type (`RiskDomain` with `NodeId`, `Score`, etc.) — the `Any` erasure forced `asInstanceOf` casts throughout register's `QueryService`
- Four design options were evaluated (A–D); Option D was selected

---

## Decision

### Option D: Generic `KnowledgeBase[D]` with Type Class Constraints

The entire data and evaluation pipeline is parameterised on a domain type `D`:

```
KnowledgeBase[D] → KnowledgeSource[D] → Model[D] → ResolvedQuery[D] → EvaluationOutput[D]
```

Type class constraints control what `D` can do:

| Type Class | Purpose | Required By |
|---|---|---|
| `DomainElement[D]` | `show: D => String` for display | Bridge, model construction |
| `DomainCodec[D]` | `fromString`, `fromNumericLiteral` for parsing | RangeExtractor, FOLBridge |
| `Ordering[D]` | Comparison operators | ComparisonAugmenter |
| `Fractional[D]` | Arithmetic operators | ArithmeticAugmenter |
| `ClassTag[D]` | Runtime array creation | HDRSampler (Fisher-Yates) |

### Why Option D

- **A (RelationValue as domain type):** Would require augmenters to output `RelationValue` — mixing KB representation with domain semantics
- **B (DomainValue ADT):** Fixed sum type, not extensible for register's custom domain
- **C (Type class without generic KB):** Bridge-layer-only generics, doesn't propagate type safety to datastore
- **D (Full generic KB):** End-to-end type safety, register defines its own `RiskDomain` with type class instances

### Backward Compatibility

`RelationValue` remains the default domain type with provided type class instances:
- `given DomainElement[RelationValue]` in companion object
- `given DomainCodec[RelationValue]` in companion object
- All existing examples and tests unchanged

## Implementation

Implemented across 5 phases (commits `4f45663`..`694eba7`):

| Phase | Scope | Commit |
|---|---|---|
| 1 | `DomainElement[D]` + `DomainCodec[D]` type classes | `4f45663` |
| 2 | Generic datastore: `KnowledgeBase[D]`, `KnowledgeSource[D]`, `RelationTuple[D]` | `1f755bd` |
| 3 | Generic bridge: `KnowledgeSourceModel`, `KnowledgeBaseModel`, augmenter decomposition | `92b378d` |
| 4+5 | Generic evaluation pipeline + query types (merged due to coupling) | `694eba7` |
| 6 | Dead code removal, ADR updates, version bump | *(uncommitted)* |

### Dead Code Removed

- `RelationValueUtil.scala` — 5 functions (`toDomainValue`, `fromDomainValue`, `toDomainSet`, `toDomainSetTyped`, `toDomainList`), all dead after Phase 3 removed their callers
- `RelationValueUtilSpec.scala` — test spec for above

### Key Technical Decisions

1. **ClassTag propagation:** `HDRSampler[A]` requires `ClassTag[A]` for Fisher-Yates array creation. This propagates through `ResolvedQuery[D: ClassTag]` → `VagueSemantics` methods.

2. **Default parameter inference:** Scala 3 cannot infer `D` in default parameters like `ModelAugmenter.identity` within generic methods. Fixed with explicit `ModelAugmenter.identity[D]` and `Map.empty[String, D]`.

3. **Augmenter decomposition (Phase 3):** `NumericAugmenter` split into `ComparisonAugmenter[D: Ordering]`, `ArithmeticAugmenter[D: Fractional]`, and `LiteralResolver[D: DomainCodec]` — each requiring only the minimum type class constraint.

## Code Smells

### ❌ Unparameterised Domain Containers

```scala
// BAD: type erasure — Any hides domain type
val model: Model[Any] = KnowledgeSourceModel.toModel(source)
val elements: Set[Any] = model.domain

// GOOD: domain type preserved end-to-end
val model: Model[D] = KnowledgeSourceModel.toModel[D](source)
val elements: Set[D] = model.domain
```

### ❌ Runtime Dispatch on Erased Types

```scala
// BAD: asInstanceOf from Any — no compiler safety
val score = element.asInstanceOf[RiskDomain].score

// GOOD: D is known at compile time
val score: Score = element.score  // element: RiskDomain
```

### ❌ Missing Type Class Constraint

```scala
// BAD: stringly-typed conversion
def show(d: Any): String = d.toString

// GOOD: type class provides semantics
def show[D: DomainElement](d: D): String = summon[DomainElement[D]].show(d)
```

---

## Consequences

- **Positive:** Compile-time type safety from KB to evaluation output — no `Any`, no `asInstanceOf`
- **Positive:** Register can define `KnowledgeBase[RiskDomain]` with custom type class instances
- **Positive:** Augmenter constraints are minimal (e.g., `Ordering` only, not full `DomainCodec`)
- **Positive:** `RelationValueUtil` eliminated — zero dead code
- **Negative:** Type parameter `[D]` adds syntactic weight to signatures
- **Negative:** `ClassTag` context bound propagates further than ideal (HDRSampler dependency)
