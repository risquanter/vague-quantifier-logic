# ADR-001: Many-Sorted Query Binding — Typed IL Compilation Phase

**Status:** Accepted  
**Date:** 2026-03-27  
**Tags:** type-safety, typed-IL, query-binding, many-sorted, type-checking

---

## Context

- A parsed query (`ParsedQuery`) carries zero sort information — variable names are bare strings, predicate and function names are untyped, literals have no declared numeric domain
- Without a type-checking phase, sort violations (wrong predicate arity, loss compared to probability, quantifier ranging over the wrong entity type) manifest as runtime exceptions or silently wrong results — they are invisible in the type system
- The distinction between *symbols declared* (catalog) and *symbols implemented* (runtime dispatcher) must be validated at startup, not discovered per-query at evaluation time
- Evaluation-time sort errors are infrastructure bugs — they mean the dispatcher or catalog was misconfigured; they must not reach user-visible error messages
- A bound typed IL (`BoundQuery`) produced once per query compilation is reusable across multiple evaluations and multiple runtime models; re-validating the catalog on every `bind` call wastes work proportional to query volume

---

## Decision

### 1. `TypeCatalog` as the Validated Sort Contract

A `TypeCatalog` value can only be produced through its validated factory — it is impossible to hold an inconsistent catalog:

```scala
// Construction always validates — returns Either on inconsistency
TypeCatalog(
  types = Set(asset, loss, prob),
  functions = Map(SymbolName("p95") -> FunctionSig(List(asset), loss)),
  predicates = Map(SymbolName("gt_loss") -> PredicateSig(List(loss, loss)))
) // Either[List[TypeCatalogError], TypeCatalog]

// Use TypeCatalog.unsafe only in tests / application startup
// where inconsistency is a programming error
TypeCatalog.unsafe(types = ..., predicates = ...)
```

### 2. `QueryBinder.bind` as the Single Compilation Step

`QueryBinder.bind` takes a catalog-guaranteed-valid `TypeCatalog` and produces
a fully annotated `BoundQuery`. It does not re-validate the catalog:

```scala
object QueryBinder:
  def bind(query: ParsedQuery, catalog: TypeCatalog): Either[List[TypeCheckError], BoundQuery]
```

Every node in `BoundQuery` carries its resolved `TypeId`. After this boundary,
no symbol lookup or sort inference is needed at evaluation time.

### 3. `BoundQuery` as the Typed Shared IL

`BoundQuery` is the canonical intermediate language for the typed evaluation path.
The `fol.typed` pipeline never evaluates `ParsedQuery` directly:

```scala
case class BoundVar(name: String, sort: TypeId)
case class BoundAtom(name: SymbolName, args: List[BoundTerm])

enum BoundTerm:
  case VarRef(v: BoundVar)
  case ConstRef(name: String, sort: TypeId)
  case FnApp(name: SymbolName, args: List[BoundTerm], resultSort: TypeId)

case class BoundQuery(
  quantifier: Quantifier,
  variable: BoundVar,     // quantified variable with declared sort
  range: BoundAtom,       // sort-checked range predicate
  scope: BoundFormula,    // sort-checked scope formula
  answerVars: List[BoundVar]
)
```

### 4. `RuntimeModel` Validates Dispatcher Coverage at Startup

A `RuntimeModel` checks that all symbols declared in the catalog have a
corresponding dispatcher implementation — at startup, not per query:

```scala
case class RuntimeModel(
  domains: Map[TypeId, Set[Value]],
  dispatcher: RuntimeDispatcher
):
  def validateAgainst(catalog: TypeCatalog): Either[List[RuntimeModelError], RuntimeModel]
```

`validateAgainst` is called once during application initialisation. If it returns
`Left`, the application must not begin serving queries.

### 5. `VagueSemantics.evaluateTyped` as the Public Canonical Entry Point

```scala
// Full typed pipeline: parse -> bind -> validate model -> evaluate
VagueSemantics.evaluateTyped(query, catalog, model)
  // Either[QueryError, EvaluationOutput[Value]]
```

## Code Smells

### ❌ Evaluating a ParsedQuery Without Binding

```scala
// BAD: sort violations reach the evaluator as runtime exceptions
TypedSemantics.evaluate(parsedQuery, model)  // does not compile by design

// GOOD: compilation phase is mandatory
QueryBinder.bind(parsedQuery, catalog).flatMap(TypedSemantics.evaluate(_, model))
```

### ❌ Validating the Catalog Inside `bind`

```scala
// BAD: catalog re-validated on every query — O(catalog-size) work per bind call
def bind(query: ParsedQuery, catalog: TypeCatalog) =
  for
    _ <- catalog.validate  // wrong: catalog is already guaranteed valid by construction
    ...

// GOOD: TypeCatalog is valid by construction; bind proceeds directly
def bind(query: ParsedQuery, catalog: TypeCatalog): Either[List[TypeCheckError], BoundQuery] =
  bindAtom(query.range, Map.empty, catalog).flatMap(...)
```

### ❌ Calling `validateAgainst` Per Query

```scala
// BAD: dispatcher coverage check in the hot query path
def handleQuery(q: ParsedQuery) =
  model.validateAgainst(catalog)  // wrong: this belongs in startup
  VagueSemantics.evaluateTyped(q, catalog, model)

// GOOD: validate once at startup; treat Left as fatal misconfiguration
val model = buildModel().validateAgainst(catalog)
  .fold(errors => sys.error(s"Dispatcher misconfigured: $errors"), identity)
```

### ❌ Using Bare String Keys for Sort Identity

```scala
// BAD: sort identity is a raw String — typos are silent
Map("Asset" -> elements)   // catalog uses TypeId("Asset"), this mismatches silently

// GOOD: opaque TypeId ensures identity consistency
val asset = TypeId("Asset")
Map(asset -> elements)
```

## Cross-ADR Relationship

This ADR covers the **first of two type boundaries** in the typed pipeline.
The second boundary — projecting `EvaluationOutput[Value]` back to consumer
domain types — is covered by ADR-013.
