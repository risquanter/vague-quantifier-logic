# ADR-001: Evaluation Path Unification

**Status:** Accepted  
**Date:** 2026-03-21  
**Tags:** architecture, evaluation, shared-IL, parser-combinator, error-handling

---

## Context

- A vague quantifier query can enter the engine through two paths: a **string parser** or a **typed DSL** — both must produce identical statistical results for the same logical query
- Without a shared intermediate representation, each path implements its own orchestration logic — a change to sampling, tolerance, or result semantics must be applied in two places
- A single PRNG strategy (HDR deterministic sampling) and a single statistical pipeline (Wilson CI, FPC sample sizing) must govern both paths to avoid silent divergence
- A type parameter on query types is only justified when it carries information that varies between real use sites — a parameter that is always instantiated to the same concrete type adds complexity without architectural benefit
- Multiple error-handling styles on the same operation (throwing, Either, custom monad) increase surface area without adding expressiveness

---

## Decision

### 1. Three Phases, One Shared IL, One Evaluator

Both entry points converge on `ResolvedQuery[D]` — a sealed type
carrying everything needed for evaluation, with no `KnowledgeSource`
dependency (see ADR-008 for the generic `D` parameter):

```scala
// fol.query.ResolvedQuery
case class ResolvedQuery[D: ClassTag](
  quantifier: VagueQuantifier,
  elements:   Set[D],
  predicate:  D => Boolean,
  params:     SamplingParams,
  hdrConfig:  HDRConfig
):
  def evaluate(): VagueQueryResult
  def evaluateWithOutput(): EvaluationOutput[D]
```

```
Phase 1 — PARSE / BUILD       Phase 2 — RESOLVE            Phase 3 — EVALUATE
───────────────────────────    ─────────────────────        ─────────────────────
String → ParsedQuery           ParsedQuery → compile →       ResolvedQuery
  or                             ResolvedQuery                 .evaluate()
Scala  → UnresolvedQuery       UnresolvedQuery → resolve →    .evaluateWithOutput()
                                 ResolvedQuery
```

### 2. Either-Only Public API with OCaml-Style Internals

Public methods return `Either[QueryError, A]`. Internal code throws
`QueryException` (OCaml `try/with Failure` style). A single
`try/catch → Either` boundary sits at each public entry point:

```scala
def parse(input: String): Either[QueryError, ParsedQuery] =
  try Right(parseUnsafe(input))
  catch case e: QueryException => Left(e.toError)

def holds(query: ParsedQuery, source: KnowledgeSource, ...): Either[QueryError, VagueQueryResult] =
  try
    val resolved = toResolved(query, source, answerTuple, params, hdrConfig)
    Right(resolved.evaluate())
  catch case e: QueryException => Left(e.toError)
```

Two boundaries, two `try/catch` blocks in the entire public surface.

### 3. `SamplingParams` Controls Evaluation Mode

No `useSampling: Boolean` on any method. `SamplingParams.exact`
(ε = 1e-6) forces n = N via the sample size formula. "Exact" is the
degenerate case of "sampled" — same code path, same evaluator:

```scala
val params = if exactMode then SamplingParams.exact else SamplingParams.default
VagueSemantics.evaluate(query, source, answerTuple, params, config)
```

### 4. One Type Flow — Justified Generics Only

Type parameters on query types are justified when they carry
information that genuinely varies between call sites. `D` varies
across consumers (`RelationValue`, `String`, register's `RiskDomain`) —
see ADR-008.

Statistical result types (`VagueQueryResult`) are always concrete —
they report proportions and confidence intervals with no domain
type dependency.

| Type | Package | Role |
|---|---|---|
| `ParsedQuery` | `fol.logic` | Parser AST — symbolic FOL |
| `UnresolvedQuery[D]` | `fol.query` | Typed DSL — domain not yet fetched |
| `ResolvedQuery[D]` | `fol.query` | **Shared IL** — domain materialized |
| `VagueQueryResult` | `fol.result` | Statistics: satisfied, proportion, CI |
| `EvaluationOutput[D]` | `fol.result` | Statistics + element sets |

### 5. Naming Policy

- **Phase names** for query types: `ParsedQuery`, `UnresolvedQuery`, `ResolvedQuery`
- **"Vague"** reserved for paper constructs: `VagueQuantifier`, `VagueQueryResult`, `VagueQueryParser`, `VagueSemantics`
- **`Query*`** for error/exception types: `QueryError`, `QueryException`

## Code Smells

### ❌ Bypassing the Shared IL

```scala
// BAD: Each path orchestrates its own evaluation
val range = RangeExtractor.extractRange(source, query, subst)
val pred  = FOLBridge.scopeToPredicate(scope, variable, source)
val count = range.count(pred)
val vq    = VagueQuantifier.fromQuantifier(query.quantifier)
vq.evaluateExact(count, range.size)  // bypasses ResolvedQuery

// GOOD: Compile to shared IL, delegate to one evaluator
val resolved = toResolved(query, source, answerTuple, params, hdrConfig)
resolved.evaluate()
```

### ❌ Multiple Error Styles on the Same Operation

```scala
// BAD: Four representations for the same failure
facade.holds(q, s)          // Either
facade.holdsEither(q, s)    // Either (redundant)
facade.holdsSafe(q, s)      // custom monad
facade.holdsUnsafe(q, s)    // throws

// GOOD: One boundary, one channel
facade.holds(q, s)          // Either[QueryError, VagueQueryResult]
```

### ❌ Gratuitous Type Parameter That Never Varies

```scala
// BAD: Type parameter that every call site instantiates identically
// (hypothetical — a parameter that adds noise without information)
val q: Query[RelationValue, ExactMode] = ...
// ExactMode is always the same everywhere — use SamplingParams instead

// GOOD: Type parameter carries real information (different D per consumer)
val resolved: ResolvedQuery[RiskDomain] = query.resolve(riskSource)
val resolved: ResolvedQuery[RelationValue] = query.resolve(defaultSource)
// D justified — different consumers use different domain types (ADR-008)
```

### ❌ Boolean Toggle for Evaluation Mode

```scala
// BAD: Two code paths based on flag
def evaluate(useSampling: Boolean): VagueQueryResult =
  if useSampling then evaluateWithSampling() else evaluateExact()

// GOOD: SamplingParams controls precision — one code path
def evaluate(): VagueQueryResult
// SamplingParams.exact forces n=N; SamplingParams.default samples
```

## Implementation

| Location | Pattern |
|---|---|
| `VagueQueryParser.parse()` | Either-only boundary; internal parser-combinators throw |
| `VagueSemantics.holds()` | Either-only boundary; delegates to `toResolved().evaluate()` |
| `VagueSemantics.evaluate()` | Either-only boundary; delegates to `toResolved().evaluateWithOutput()` |
| `VagueSemantics.toResolved()` | Steps A1–A5: range extraction, scope compilation, quantifier conversion, IL construction |
| `ResolvedQuery.evaluate()` | One evaluator for both paths — calls `ProportionEstimator` |
| `ResolvedQuery.evaluateWithOutput()` | Same evaluator + element sets → `EvaluationOutput` |
| `UnresolvedQuery.resolve(source)` | Typed DSL resolution: fetches domain from `KnowledgeSource` → `ResolvedQuery` |
| `RangeExtractor.extractRange()` | FOL pattern resolver — Either boundary, internal throws |
| `FOLBridge.scopeToPredicate()` | Compiles FOL formula → `RelationValue => Boolean` closure |
| `ProportionEstimator.estimateWithSampling()` | Single-pass: filter + count + Wilson CI |
| `ProportionEstimator.estimateFromCount()` | From pre-counted integers — no re-iteration |
