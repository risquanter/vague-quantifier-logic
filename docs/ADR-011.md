# ADR-011: DSL Removal — IL-Direct Construction

**Status:** Accepted  
**Date:** 2026-03-25  
**Tags:** architecture, dsl, simplification, evaluation-path

---

## Context

- The typed DSL (`Query` builder, `UnresolvedQuery`, `DomainSpec`, `Predicates`) provided a fluent API for programmatic query construction, converging on the shared IL (`ResolvedQuery`) via `.resolve(source)`
- The DSL added marginal value over the IL: the fluent syntax was cosmetic (method chaining), and deferred resolution was its only structural contribution
- Two convergence incidents occurred where semantic decisions (empty-range handling, unknown-relation boundary checks) had to be applied to both the DSL and FOL entry points independently — each instance creating silent divergence risk
- `DomainSpec.ActiveDomain` made the DSL strictly richer than the FOL grammar (Definition 2 requires a named range predicate), creating a semantic surface mismatch (TD-002)
- ADR-001 established `ResolvedQuery` as the shared IL — the DSL was one of two paths into that IL, but a thin factory method on `ResolvedQuery` itself can serve the same purpose with zero divergence risk

---

## Decision

### 1. Remove the Typed DSL Entirely

Delete:
- `Query` builder (`Query.quantifier().over().where()`)
- `UnresolvedQuery[D]` case class
- `DomainSpec` enum (`Relation`, `ActiveDomain`)
- `Predicates` object (`inRelation`, `hasRelation`, `relatedTo`)
- `source.execute()` / `source.executeWithOutput()` extension methods

### 2. Add `ResolvedQuery.fromRelation` Factory

A single factory method on `ResolvedQuery` companion replaces the
DSL's `resolve()`:

```scala
object ResolvedQuery:
  def fromRelation[D: ClassTag](
    source: KnowledgeSource[D],
    relationName: RelationName,
    position: Int = 0,
    quantifier: VagueQuantifier,
    predicate: D => Boolean,
    params: SamplingParams = SamplingParams.exact,
    hdrConfig: HDRConfig = HDRConfig.default
  ): Either[QueryError, ResolvedQuery[D]]
```

This performs the same `hasRelation` boundary check as the former
`UnresolvedQuery.resolve()`, then materializes domain elements and
returns a sealed `ResolvedQuery`.

### 3. No `fromActiveDomain` Factory

`DomainSpec.ActiveDomain` quantified over the union of all stored
entities — schema-dependent and fragile.  Any legitimate use case is
better served by an explicit unary relation (e.g. `server("serverA")`)
queried via `fromRelation`.

The `KnowledgeSource.activeDomain` method remains in the core API
(used by FOL evaluation), but no programmatic entry point encourages
its use as a quantification domain.

### 4. Drop Predicate Helpers

`Predicates.inRelation`, `Predicates.hasRelation`, `Predicates.relatedTo`
were one-line wrappers over `KnowledgeSource.query` / `.contains`.
Callers write inline closures:

```scala
// "borders(x, France)" as scope predicate:
val pred: D => Boolean =
  d => source.query(RelationName("borders"), List(Some(d), Some(target))).nonEmpty
```

---

## Consequences

- **Positive:** Single entry point for programmatic queries — zero divergence risk between paths
- **Positive:** TD-002 (ActiveDomain semantic smell) closed — `DomainSpec.ActiveDomain` no longer exists
- **Positive:** TD-001 (duplicated boundary checks) reduced — `hasRelation` check now co-located in `ResolvedQuery.fromRelation` and `RangeExtractor.extractRange`
- **Positive:** ~636 lines of source code and ~1050 lines of test code removed
- **Negative:** Programmatic queries are slightly more verbose (no fluent chain)
- **Negative:** ~66 fewer tests (77 DSL tests removed, ~21 translated to IL)
