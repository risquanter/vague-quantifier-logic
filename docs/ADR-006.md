# ADR-006: Domain Type Erasure in KB-Backed Models

**Status:** Accepted  
**Date:** 2026-03-23  
**Tags:** type-safety, model-theory, domain, trade-off

---

## Context

- FOL model theory requires a single domain type `D` for all elements — `Model[D]`, `Interpretation[D]`, `Valuation[D]` are all parameterised on the same `D`
- `RelationValue` is already a proper Scala 3 ADT: `enum RelationValue: case Const(String); case Num(Int)` — type information exists at the KB boundary
- `RelationValueUtil.toDomainValue` unwraps `RelationValue` into `Any`, discarding the sum type — this is where type safety is lost
- KB-backed models serve as `Model[Any]` because the domain contains both `String` and `Int` values in a single `Set[Any]`
- `NumericAugmenter.toDouble` pattern-matches on `Any` at runtime — non-exhaustive, no compiler assistance on missing cases

---

## Decision

### 1. Accept `Model[Any]` for KB-Backed Models (Current State)

KB-backed models use `Model[Any]` with runtime type dispatch in
`NumericAugmenter.toDouble` and `RelationValueUtil.fromDomainValue`.

This is a **pragmatic shortcut, not a fundamental FOL constraint**.
FOL requires a single domain type, but that type could be `RelationValue`
(the existing ADT) or a dedicated `DomainValue` sum type — both would
preserve exhaustiveness checking.

```scala
// Current: type information discarded
def toDomainValue(rv: RelationValue): Any = rv match
  case RelationValue.Const(name) => name  // String
  case RelationValue.Num(value) => value  // Int
// → Model[Any] — no compiler help on type mismatches
```

### 2. Known Alternative: `RelationValue` as Domain Type

The domain type could remain `RelationValue` throughout, avoiding
the `Any` erasure entirely:

```scala
// Alternative: RelationValue preserved as domain type
def toModel(source: KnowledgeSource): Model[RelationValue] =
  val domainElements: Set[RelationValue] = source.activeDomain
  // ...

// toDouble becomes exhaustive:
def toDouble(v: RelationValue): Double = v match
  case RelationValue.Const(s) => s.toDoubleOption.getOrElse(...)
  case RelationValue.Num(i)   => i.toDouble
  // Compiler verifies all cases covered
```

### 3. Known Alternative: Dedicated `DomainValue` ADT

A richer sum type that extends `RelationValue` with augmenter-produced
values (`Double`, `Long`, `BigDecimal`):

```scala
// Alternative: explicit sum type for all domain values
enum DomainValue:
  case Str(v: String)
  case Num(v: Double)
  case Integer(v: Int)

// Pattern matches are exhaustive, compiler warns on missing cases
```

### 4. Why `Any` Is Accepted For Now

The retrofit cost is high — `toDomainValue`, `fromDomainValue`,
`KnowledgeSourceModel`, `KnowledgeBaseModel`, `EvaluationContext`,
`FOLBridge`, all test fixtures, and every consumer constructing
`Interpretation` directly would need updating. The benefit is real
(exhaustiveness checking) but the runtime behaviour is identical —
the same type dispatch happens, just with compiler assistance.

This is a candidate for a future refactoring pass if `DomainValue`
variants proliferate beyond `String`/`Int`/`Double`.

## Code Smells

### ❌ Assuming Domain Values Are Always String

```scala
// BAD: unchecked cast, fails on Num values
val name = domainValue.asInstanceOf[String]

// GOOD: pattern match with all cases
domainValue match
  case s: String => handleString(s)
  case i: Int    => handleInt(i)
  case d: Double => handleDouble(d)
  case other     => throw new Exception(
    s"Unexpected domain value: $other (${other.getClass.getName})")
```

### ❌ Adding New Types Without Updating `toDouble`

```scala
// BAD: new type flows into domain without conversion support
val domain = Set[Any]("alice", 42, 3.14f)  // Float not handled

// GOOD: ensure NumericAugmenter.toDouble covers all numeric types
// Current coverage: Double, Int, Long, BigDecimal, numeric String
```

### ❌ Relying on `ClassCastException` Instead of Pattern Match

```scala
// BAD: implicit cast
val d: Double = value.asInstanceOf[Double]

// GOOD: explicit dispatch with clear error
NumericAugmenter.toDouble(value)  // handles Int, Long, BigDecimal, String
```

## Implementation

| Location | Current State |
|---|---|
| `RelationValueUtil.toDomainValue` | `RelationValue → Any` — erasure point |
| `KnowledgeSourceModel.toModel` | Returns `Model[Any]` |
| `NumericAugmenter.toDouble` | `Any → Double` runtime dispatch |
| `FOLBridge.scopeToPredicate` | Operates on `Model[Any]` |
| `FOLSemantics.integerModel` | `Model[Int]` — type-safe, no erasure |
