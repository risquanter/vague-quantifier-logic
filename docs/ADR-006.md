# ADR-006: ADT Encoding — `enum` for Pure-Data Sum Types, `sealed trait` for Behavioural Hierarchies

**Status:** Accepted  
**Date:** 2026-04-04  
**Tags:** scala3, adt, coding-style, type-safety, maintainability

---

## Context

- Scala 3 provides two overlapping mechanisms for closed sum types: `enum` and `sealed trait` + `case class`.
- `sealed trait` + top-level `case class` places variant names directly in the enclosing package namespace, increasing the risk of name collisions and making it impossible to tell at a glance which variants belong together.
- `enum` scopes variants inside the enum's companion, making the grouping structurally explicit and eliminating ambiguity.
- Some sum types carry per-variant abstract behaviour (methods that differ in each case); others carry only data (constructor parameters and no variant-specific logic).
- Mixing the wrong encoding with the wrong intent produces code that is either unnecessarily verbose or unnecessarily flat.

---

## Decision

### 1. `enum` for pure-data sum types

Use `enum` when all variants are distinguished only by constructor parameters — no abstract methods, no per-variant method bodies.

```scala
// GOOD — closed set of data shapes, no per-variant logic
enum TypeDecl:
  case DomainType(id: TypeId)
  case ValueType(id: TypeId)

  def typeId: TypeId = this match  // single shared derivation is fine
    case DomainType(id) => id
    case ValueType(id)  => id

enum LiteralValue:
  case IntLiteral(value: Long)
  case FloatLiteral(value: Double)
  case TextLiteral(value: String)
```

Variants are accessed as `TypeDecl.DomainType(...)` and `LiteralValue.IntLiteral(...)`.  
Within the same package, import with `import TypeDecl.*` / `import LiteralValue.*` for short names.  
From other packages, prefer `import pkg.TypeDecl.{DomainType, ValueType}`.

### 2. `sealed trait` for behavioural hierarchies

Use `sealed trait` when variants override abstract methods declared on the parent, or when variant bodies are too large to fit cleanly inside an `enum`.

```scala
// GOOD — each variant has unique method implementations
sealed trait VagueQuantifier:
  def contains(p: Double): Boolean
  def membershipDegree(p: Double): Double

case class Approximately(target: Double, tolerance: Double = 0.1) extends VagueQuantifier:
  def contains(p: Double): Boolean = math.abs(p - target) <= tolerance
  def membershipDegree(p: Double): Double = ???

case class AtLeast(threshold: Double, tolerance: Double = 0.0) extends VagueQuantifier:
  def contains(p: Double): Boolean = p >= threshold - tolerance
  def membershipDegree(p: Double): Double = ???
```

### 3. Derived members in `enum`

When all cases share a logical accessor, declare it as a `def` in the `enum` body using pattern matching. Do not use it as a direct constructor parameter name (avoids name-clash compile errors).

```scala
enum BoundTerm:
  case VarRef(v: BoundVar)
  case ConstRef(sourceText: String, sort: TypeId, raw: LiteralValue)
  case FnApp(name: SymbolName, args: List[BoundTerm], resultSort: TypeId)

  def sort: TypeId = this match
    case VarRef(v)         => v.sort
    case ConstRef(_, s, _) => s
    case FnApp(_, _, s)    => s
```

### 4. Imports for `enum` variants

Callers within the same package add a local import; callers from other packages use the qualified path or a named import — never open a new wildcard import from a foreign package.

```scala
// Same package
import TypeDecl.*
import LiteralValue.*

// Other package
import fol.typed.TypeDecl.{DomainType, ValueType}
import fol.typed.LiteralValue.{IntLiteral, FloatLiteral}
```

## Code Smells

### ❌ Top-level case classes that form a conceptual group

```scala
// BAD — variants pollute the package namespace; grouping is invisible
sealed trait LiteralValue
case class IntLiteral(value: Long)    extends LiteralValue
case class FloatLiteral(value: Double) extends LiteralValue

// GOOD
enum LiteralValue:
  case IntLiteral(value: Long)
  case FloatLiteral(value: Double)
```

### ❌ `enum` for types with per-variant abstract behaviour

```scala
// BAD — each case needs distinct method bodies; enum body becomes unwieldy
enum VagueQuantifier:
  case Approximately(target: Double, tolerance: Double)
    def contains(p: Double): Boolean = math.abs(p - target) <= tolerance  // not valid in simple enum
  ...

// GOOD  — sealed trait lets each case class carry its own implementation
sealed trait VagueQuantifier:
  def contains(p: Double): Boolean
```

### ❌ Accessing enum variants without import or qualification

```scala
// BAD — DomainType looks like a top-level class; reader cannot trace its origin
val t: TypeDecl = DomainType(TypeId("Asset"))

// GOOD
import TypeDecl.*
val t: TypeDecl = DomainType(TypeId("Asset"))    // import visible at file top
// or
val t: TypeDecl = TypeDecl.DomainType(TypeId("Asset"))   // qualified, no import needed
```

## Cross-ADR Relationship

ADR-014 uses `TypeDecl` (`DomainType` / `ValueType`) to partition a catalog's declared types into quantifiable and non-quantifiable. ADR-015 uses `LiteralValue` (`IntLiteral` / `FloatLiteral` / `TextLiteral`) as the injection boundary for inline literals. Both ADTs are pure-data sum types and are encoded as `enum` per this ADR. The semantic contracts in ADR-014 and ADR-015 are unaffected by encoding choice.
