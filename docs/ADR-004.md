# ADR-004: Tagless Initial Architecture

**Status:** Accepted  
**Date:** 2026-03-21  
**Tags:** architecture, tagless-initial, ADT, OCaml, separation-of-concerns

---

## Context

- A logic engine operates on recursive tree structures (formulas, terms, queries) that benefit from exhaustive pattern matching — adding new operations must not require modifying the data types
- OCaml's module system separates types from functions — a Scala port preserving this separation avoids coupling data layout to operation sets
- OO-style methods on data types conflate "what the data is" with "what can be done to it" — printing, evaluation, optimization, and serialization are independent concerns that evolve at different rates
- The FOL foundation must be reusable independently of the vague quantifier extension — layering requires that the extension imports the foundation, not the other way around

---

## Decision

### 1. ADTs for Data, Objects for Operations

Data structures are enums and case classes. Operations live in
companion-free top-level objects. No methods on data types beyond
structural accessors:

```scala
// logic/Formula.scala — data only
enum Formula[A]:
  case Atom(a: A)
  case Not(f: Formula[A])
  case And(f: Formula[A], g: Formula[A])
  case Or(f: Formula[A], g: Formula[A])
  case Imp(f: Formula[A], g: Formula[A])
  case Iff(f: Formula[A], g: Formula[A])
  case Forall(x: String, f: Formula[A])
  case Exists(x: String, f: Formula[A])

// semantics/FOLSemantics.scala — operations
object FOLSemantics:
  def holds[D](formula: Formula[FOL], model: Model[D], valuation: Valuation[D]): Boolean

// printer/FOLPrinter.scala — separate operation
object FOLPrinter:
  def printFormula(formula: Formula[FOL], prec: Int = 0): String
```

### 2. Two-Layer Architecture: FOL Foundation + Vague Extension

The FOL layer (`logic/`, `parser/`, `semantics/`, `printer/`, `lexer/`)
provides general-purpose first-order logic. The vague layer (`fol/`)
extends it with probabilistic quantifiers. The vague layer imports FOL;
FOL never imports vague:

```
┌──────────────────────────────────────────────────────┐
│  fol/  (vague quantifier extension)                  │
│  ├── parser/     VagueQueryParser                    │
│  ├── semantics/  VagueSemantics, RangeExtractor,     │
│  │               ScopeEvaluator, DomainExtraction    │
│  ├── query/      ResolvedQuery, UnresolvedQuery      │
│  ├── sampling/   HDRSampler, ProportionEstimator     │
│  ├── bridge/     FOLBridge, KnowledgeSourceModel     │
│  ├── datastore/  KnowledgeBase, KnowledgeSource      │
│  ├── result/     VagueQueryResult, EvaluationOutput  │
│  └── quantifier/ VagueQuantifier                     │
├──────────────────────────────────────────────────────┤
│  Foundation (pure FOL — no vague imports)             │
│  ├── logic/      Formula, Term, FOL, Model           │
│  ├── parser/     Combinators, FormulaParser,         │
│  │               FOLAtomParser, TermParser            │
│  ├── lexer/      Lexer                               │
│  ├── semantics/  FOLSemantics, EvaluationContext      │
│  └── printer/    FOLPrinter, PrinterUtil             │
└──────────────────────────────────────────────────────┘
```

### 3. Bridge Layer for KB → FOL Model Translation

`KnowledgeBase[D]` stores typed tuples; FOL evaluation requires `Model[D]`.
Bridge utilities translate between these representations while preserving
the domain type `D` end-to-end (see ADR-008):

```scala
// fol/bridge/KnowledgeBaseModel
object KnowledgeBaseModel:
  def toModel[D: DomainElement](kb: KnowledgeBase[D]): Model[D]

// fol/bridge/KnowledgeSourceModel
object KnowledgeSourceModel:
  def toModel[D: DomainElement](source: KnowledgeSource[D]): Model[D]
```

### 4. `EvaluationContext` Wraps Model + Valuation

Instead of threading `model` and `valuation` separately through every
call, `EvaluationContext` bundles them. Extension methods constrain
KB-specific operations to `EvaluationContext[Any]`:

```scala
// semantics/EvaluationContext
case class EvaluationContext[D](model: Model[D], valuation: Valuation[D]):
  def holds(formula: Formula[FOL]): Boolean
  def withBinding(variable: String, value: D): EvaluationContext[D]

extension (ctx: EvaluationContext[Any])
  def holdsWithRelationValue(formula: Formula[FOL], variable: String, value: RelationValue): Boolean
```

## Code Smells

### ❌ Methods on Data Types

```scala
// BAD: Operation coupled to data
enum Formula[A]:
  case And(f: Formula[A], g: Formula[A]):
    def holds(model: Model[Any]): Boolean = ...  // printing, optimization also here?

// GOOD: Data is inert, operations are separate
enum Formula[A]:
  case And(f: Formula[A], g: Formula[A])

object FOLSemantics:
  def holds[D](formula: Formula[FOL], model: Model[D], valuation: Valuation[D]): Boolean =
    formula match
      case And(f, g) => holds(f, model, valuation) && holds(g, model, valuation)
```

### ❌ Vague Layer Leaking into FOL

```scala
// BAD: FOL layer knows about vague quantifiers
object FOLSemantics:
  def holdsVague(query: ParsedQuery, kb: KnowledgeBase): Boolean = ...

// GOOD: Vague layer calls FOL, not the other way
object VagueSemantics:
  def holds(...) = ... FOLSemantics.holds(scope, model, valuation) ...
```

### ❌ Scattered Type Conversions

```scala
// BAD: manual domain-to-string conversion duplicated across files
val name: String = element match
  case RelationValue.Const(n) => n
  case RelationValue.Num(v)   => v.toString

// GOOD: use the DomainElement type class (ADR-008)
val name: String = summon[DomainElement[D]].show(element)
```

## Implementation

| Location | Pattern |
|---|---|
| `logic/Formula.scala`, `logic/Term.scala` | Pure ADTs — no methods beyond accessors |
| `semantics/FOLSemantics.holds()` | Tarski evaluation via pattern matching |
| `printer/FOLPrinter.printFormula()` | Precedence-aware printing (uses `PrinterUtil`) |
| `fol/bridge/KnowledgeSourceModel` | KB → `Model[D]` conversion (ADR-008) |
| `fol/bridge/KnowledgeBaseModel` | KB → `Model[D]` direct (ADR-008) |
| `semantics/EvaluationContext` | `Model + Valuation` bundle with extension methods |
