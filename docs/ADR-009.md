# ADR-009: Symmetric Relation Support via Schema Metadata

**Status:** Accepted  
**Date:** 2026-03-25  
**Tags:** datastore, relations, symmetry, knowledge-base, schema

---

## Context

- Relations in FOL / Tarski semantics are sets of **ordered tuples** ‚Äî `R(a,b)` and `R(b,a)` are independent truth values
- Many real-world relationships are naturally symmetric: "borders", "knows", "communicates_with"
- Without infrastructure support, symmetry must be encoded by manually inserting both `(A,B)` and `(B,A)` ‚Äî error-prone with no compile-time or runtime safety net
- `Predicates.relatedTo` hardcodes argument order (entity = pos 0, target = pos 1), making reverse-direction queries impossible without a separate mirrored relation
- Five design options were evaluated (A‚ÄìE); Option B was selected

---

## Decision

### 1. Schema Property Enum

Relations declare structural properties via `RelationProperty`:

```scala
enum RelationProperty:
  case Symmetric   // R(a,b) ‚Üî R(b,a), binary only

case class Relation(
  name: String,
  arity: Int,
  properties: Set[RelationProperty] = Set.empty
)
```

Binary-only constraint enforced at construction:

```scala
require(
  !(properties.contains(RelationProperty.Symmetric) && !isBinary),
  s"Symmetric property requires binary relation, but $name has arity $arity"
)
```

### 2. Data-Load-Time Materialisation

`KnowledgeBase.addFact` auto-inserts the reverse tuple for symmetric relations:

```scala
def addFact(relationName: String, tuple: RelationTuple[D]): KnowledgeBase[D] =
  // ... arity validation ...
  val withForward = currentFacts + tuple
  val updated =
    if relation.isSymmetric then
      withForward + RelationTuple(tuple.values.reverse)
    else
      withForward
  copy(facts = facts + (relationName -> updated))
```

Materialisation is **idempotent** ‚Äî sets absorb duplicates. Self-referential tuples `(A,A)` produce no extra entry.

### 3. Factory Method

Convenience constructor for symmetric binary relations:

```scala
object Relation:
  def symmetricBinary(name: String): Relation =
    Relation(name, 2, Set(RelationProperty.Symmetric))
```

### 4. Zero Downstream Changes

Because materialisation happens at insert time, all downstream layers see both directions without modification:

- `KnowledgeSource.contains` / `.query` ‚Äî unchanged
- `KnowledgeBaseModel.toModel` / `KnowledgeSourceModel.toModel` ‚Äî unchanged
- `Predicates.relatedTo` / `.hasRelation` / `.inRelation` ‚Äî unchanged
- `FOLSemantics.holds` ‚Äî unchanged
- `VagueSemantics.holds` (string API) ‚Äî unchanged

## Code Smells

### ‚ùå Manual Symmetric Duplication

```scala
// BAD: caller responsible for both directions
kb.addFact("borders", binary("France", "Germany"))
kb.addFact("borders", binary("Germany", "France")) // easy to forget
```

```scala
// GOOD: schema declares symmetry, reverse materialised automatically
val kb = KnowledgeBase.empty[RelationValue]
  .addRelation(Relation.symmetricBinary("borders"))
  .addFact("borders", binary("France", "Germany"))
// borders(Germany, France) exists automatically
```

### ‚ùå Separate Inverse Relations for Symmetric Concepts

```scala
// BAD: two relations encoding the same symmetric relationship
kb.addRelation(Relation.binary("borders"))
kb.addRelation(Relation.binary("bordered_by"))
kb.addFact("borders", binary("France", "Germany"))
kb.addFact("bordered_by", binary("Germany", "France"))
```

```scala
// GOOD: single symmetric relation
kb.addRelation(Relation.symmetricBinary("borders"))
kb.addFact("borders", binary("France", "Germany"))
```

### ‚ùå Marking Non-Binary Relations as Symmetric

```scala
// BAD: compile-time type allows it, runtime rejects ‚Äî caught by require()
Relation("tagged", 1, Set(RelationProperty.Symmetric))
// throws: "Symmetric property requires binary relation"
```

```scala
// GOOD: use the factory
Relation.symmetricBinary("borders") // enforces arity = 2
```

|----------|---------|

| `KnowledgeBase.scala` ‚Äî `addFact` | Auto-materialisation of reverse tuple |
| `SymmetricRelationSpec.scala` | 22 unit tests: schema, materialisation, query, builder, generic `D` |
| `SymmetricRelationIntegrationSpec.scala` | 8 integration tests: DSL Predicates, Query DSL, FOL bridge, DSL‚ÜîFOL consistency |

## Alternatives Rejected

### A ‚Äî Document Only (no code change)
- **What:** Document directional semantics; users manage symmetry manually
- **Why rejected:** Error-prone ‚Äî forgetting the reverse tuple produces silent logic bugs with no compile-time or runtime guard

### C ‚Äî Query-Time Symmetric Lookup
- **What:** `KnowledgeSource.query` tries the reversed pattern when the forward pattern fails
- **Why rejected:** Adds a runtime branch to every query; breaks the simple set-membership model; harder to reason about correctness; performance cost on every lookup

## Future Exploration (non-binding)

> These are **not decisions** ‚Äî they are recorded ideas for potential future work.

- **Inverse relation declarations (Option D):** Allow `Relation` to declare `inverseOf: Option[String]` for named inverse relations (OWL `owl:inverseOf`). `addFact("has_risk", (A,B))` would auto-insert `("affected_by", (B,A))`. Symmetric is the degenerate case where inverse = self.
- **Predicate combinators (Option E):** A `Predicates.symmetrically(pred)` combinator for query-time symmetry on ad-hoc predicates without schema changes. Useful for one-off queries where modifying the schema is undesirable.
