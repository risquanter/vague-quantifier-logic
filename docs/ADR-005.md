# ADR-005: Model Augmentation via Functional Composition

**Status:** Accepted  
**Date:** 2026-03-23  
**Tags:** model-theory, composition, monoid, augmentation, extensibility

---

## Context

- FOL model theory requires a single `Interpretation[D]` that resolves **all** symbols — KB relations, numeric comparisons, arithmetic, and domain-specific functions — with no distinction by origin
- Harrison's `termval` dispatches `func f args` as a single total function; the evaluator does not distinguish "KB-backed" from "computed" interpretations
- `Interpretation` composition must form a **monoid**: an identity augmenter that changes nothing, and an associative binary operation so augmenters can be chained in any grouping
- KB-backed models provide only relation-membership predicates and identity constants — comparison predicates (`>`, `<`), numeric literal resolution, and consumer-defined functions require injection without modifying the KB
- The extension mechanism must be **additive**: existing API signatures remain source-compatible via default parameters

---

## Decision

### 1. `Interpretation` Composition — Monoid over Map Merge

`Interpretation[D]` gains combinators that merge function and predicate
maps. Map merge (`++`) with right-bias is a monoid: empty map is the
identity element, `++` is associative, and the last writer wins on
key collisions — giving later augmenters override priority:

```scala
// semantics/FOLSemantics.scala — on Interpretation[D]
def withFunctions(extra: Map[String, List[D] => D]): Interpretation[D] =
  copy(funcInterp = funcInterp ++ extra)

def withPredicates(extra: Map[String, List[D] => Boolean]): Interpretation[D] =
  copy(predInterp = predInterp ++ extra)

def withDomain(extra: Set[D]): Interpretation[D] =
  copy(domain = Domain(domain.elements ++ extra))

def combine(other: Interpretation[D]): Interpretation[D] =
  Interpretation(
    Domain(domain.elements ++ other.domain.elements),
    funcInterp ++ other.funcInterp,
    predInterp ++ other.predInterp
  )
```

**Monoid laws hold** because `Map.++` and `Set.++` are both associative
with empty as identity:

- `i.combine(empty) ≡ i` and `empty.combine(i) ≡ i` (identity)
- `(a.combine(b)).combine(c) ≡ a.combine(b.combine(c))` (associativity)

### 2. `ModelAugmenter` — Endomorphism Monoid (Case Class Wrapper)

`ModelAugmenter[D]` wraps `Model[D] => Model[D]` in a nominal case
class. Endomorphisms form a **monoid under composition** (`andThen`)
with `identity` as the unit. The wrapper provides:

- A nominal type that can host type class instances (consumers using
  ZIO Prelude, Cats, or similar can provide `given Identity[ModelAugmenter[Any]]`
  in a 3-line adapter — no library dependency required)
- `andThen` / `compose` methods directly on the type
- Structural equality for testing

```scala
// semantics/ModelAugmenter.scala
case class ModelAugmenter[D](run: Model[D] => Model[D]):
  def apply(model: Model[D]): Model[D] = run(model)
  def andThen(that: ModelAugmenter[D]): ModelAugmenter[D] =
    ModelAugmenter(m => that.run(this.run(m)))
  def compose(that: ModelAugmenter[D]): ModelAugmenter[D] =
    ModelAugmenter(m => this.run(that.run(m)))

object ModelAugmenter:
  def identity[D]: ModelAugmenter[D] = ModelAugmenter(m => m)

  /** Compose left-to-right. Associative by construction. */
  def combine[D](first: ModelAugmenter[D], second: ModelAugmenter[D]): ModelAugmenter[D] =
    first andThen second

  /** Lift function/predicate maps into an augmenter. */
  def fromFunctions[D](fns: Map[String, List[D] => D]): ModelAugmenter[D] =
    ModelAugmenter(m => Model(m.interpretation.withFunctions(fns)))

  def fromPredicates[D](preds: Map[String, List[D] => Boolean]): ModelAugmenter[D] =
    ModelAugmenter(m => Model(m.interpretation.withPredicates(preds)))
```

Monoid laws hold because function composition is associative and
`identity` is the neutral element.

**Consumer interop (e.g., ZIO Prelude in register):**

```scala
// register-side — 3 lines, no fol-engine changes needed
import zio.prelude.Identity
given Identity[ModelAugmenter[Any]] with
  def identity: ModelAugmenter[Any] = ModelAugmenter.identity
  def combine(l: => ModelAugmenter[Any], r: => ModelAugmenter[Any]): ModelAugmenter[Any] =
    l andThen r
```

fol-engine provides the monoid *operations* (`identity`, `andThen`);
the consumer provides the type class *instance* in its own framework.
No ZIO Prelude / Cats / Scalaz dependency in fol-engine.

### 3. `getFunction`/`getPredicate` Fallback Chain

Numeric literal resolution requires handling an **open class** of names
(any digit string). Rather than pre-registering every number, a
fallback function provides dynamic dispatch when the map lookup misses.
Fallbacks compose via `Option.orElse` — a monoid on `Option[A]` with
`None` as identity:

```scala
// semantics/FOLSemantics.scala — on Interpretation[D]
def withFunctionFallback(
  fallback: String => Option[List[D] => D]
): Interpretation[D] =
  val outer = this
  new Interpretation[D](domain, funcInterp, predInterp):
    override def getFunction(name: String): List[D] => D =
      funcInterp.get(name)
        .orElse(fallback(name))
        .getOrElse(_ => throw new Exception(s"Uninterpreted function: $name"))
```

Multiple fallbacks chain:
`fb1.orElse(fb2).orElse(fb3)` — associative, `_ => None` is identity.

**Why `getPredicate` has no fallback:** Predicates in practice form a
*closed set* at composition time — you know all comparison operators
(`>`, `<`, `>=`, `<=`, `=`) when the augmenter is constructed. Unlike
functions, where numeric literals form an *open class* of names (any
digit string like `"42"`, `"3.14"`), there is no known use case for a
predicate family whose names are not enumerable upfront.

Recommended patterns for the likely cases:

- **Comparison predicates**: Use `NumericAugmenter.augmenter`, which
  registers all five via `withPredicates` (Map merge).
- **Domain-specific predicates** (e.g., `isActive`, `isCritical`): Use
  `ModelAugmenter.fromPredicates(Map("isActive" -> ...))` — names known
  at augmenter construction time.
- **Predicate families** (hypothetical: `contains_X` for any X): Use
  `withPredicates` inside a `ModelAugmenter` that inspects the model's
  domain at augmentation time and generates the full map.

Adding `predFallback` would be a symmetric 4-line change mirroring
`funcFallback` if a genuine open-class predicate scenario arises.

### 4. Built-In Numeric Augmenter

Comparison predicates and numeric literal resolution are generic
infrastructure provided by fol-engine as a reusable augmenter.
Domain-specific functions (e.g., `p95`, `lec`) are consumer-supplied:

```scala
// fol/bridge/NumericAugmenter.scala
object NumericAugmenter:
  def augmenter: ModelAugmenter[Any] = ModelAugmenter(model =>
    val preds = Map[String, List[Any] => Boolean](
      ">"  -> { case List(a, b) => toDouble(a) > toDouble(b) },
      "<"  -> { case List(a, b) => toDouble(a) < toDouble(b) },
      ">=" -> { case List(a, b) => toDouble(a) >= toDouble(b) },
      "<=" -> { case List(a, b) => toDouble(a) <= toDouble(b) },
      "="  -> { case List(a, b) => toDouble(a) == toDouble(b) },
    )
    val withPreds = model.interpretation.withPredicates(preds)
    Model(withPreds.withFunctionFallback(numericLiteral))
  )

  private def numericLiteral(name: String): Option[List[Any] => Any] =
    name.toDoubleOption.map(d => _ => d)

  private def toDouble(v: Any): Double = v match
    case d: Double     => d
    case i: Int        => i.toDouble
    case l: Long       => l.toDouble
    case bd: BigDecimal => bd.toDouble
    case s: String     => s.toDouble
```

### 5. API Threading — Default `identity`, Additive Overloads

The augmenter parameter threads through the call chain with
`identity` as default — all existing call sites compile unchanged:

```scala
// FOLBridge.scopeToPredicate — augmenter parameter, generic D
def scopeToPredicate[D: DomainElement](
  formula: Formula[FOL], variable: String,
  source: KnowledgeSource[D],
  answerTuple: Map[String, D] = Map.empty[String, D],
  modelAugmenter: ModelAugmenter[D] = ModelAugmenter.identity[D]
): D => Boolean =
  val model = modelAugmenter(KnowledgeSourceModel.toModel(source))
  // ... closure over augmented model
```

Consumers compose augmenters at the call site:

```scala
// Register composes comparison infra + domain functions
val augmenter = ComparisonAugmenter.augmenter[RiskDomain] andThen riskTreeAugmenter
VagueSemantics.evaluate(query, source, modelAugmenter = augmenter)
```

## Code Smells

### ❌ Mutating Interpretation Maps In Place

```scala
// BAD: side-effecting map modification
val interp = model.interpretation
interp.funcInterp.asInstanceOf[mutable.Map] += ("p95" -> myFn)

// GOOD: pure composition via copy-on-merge
val augmented = model.interpretation.withFunctions(Map("p95" -> myFn))
Model(augmented)
```

### ❌ Bypassing the Augmenter by Rebuilding Model

```scala
// BAD: reconstructing Model manually, duplicating conversion logic
val model = KnowledgeSourceModel.toModel(source)
val hacked = Model(Interpretation(model.domain, myFuncs, myPreds))
// ... manually call FOLSemantics.holds

// GOOD: use the augmenter parameter
VagueSemantics.holds(query, source,
  modelAugmenter = NumericAugmenter.augmenter)
```

### ❌ Hardcoding Numeric Infrastructure in Each Consumer

```scala
// BAD: every consumer redefines comparison predicates
val myAugmenter = ModelAugmenter[Any](model =>
  Model(model.interpretation.withPredicates(Map(
    ">" -> { case List(a, b) => /* same as NumericAugmenter */ }
  )))
)

// GOOD: compose the built-in augmenter
val myAugmenter = NumericAugmenter.augmenter andThen myDomainAugmenter
```

### ❌ Using Inheritance to Override `getFunction` Directly

```scala
// BAD: anonymous subclass for one-off override
val interp = new Interpretation(...) {
  override def getFunction(name: String) = /* custom fallback */
}

// GOOD: withFunctionFallback composes cleanly
val interp = base.withFunctionFallback(name =>
  name.toLongOption.map(n => (_: List[Any]) => n)
)
```

## Alternatives Rejected

### Precomputed KB Facts

**What**: Consumers precompute derived values (e.g., quantiles) and
store them as relational facts in the KnowledgeBase.

**Why rejected**: Fails at three independent levels. (1) FOL syntax
`>(p95(x), 5000000)` parses `p95(x)` as a *function application* that
must return a value — a KB fact can only confirm membership, not produce
a value. (2) Comparison predicates `>`, `<` are absent from KB-backed
models regardless of stored data. (3) `RelationValue.Num` wraps `Int`,
insufficient for `Long`/`Double` simulation outputs.

### Direct Model Parameter (No Augmenter)

**What**: Pass a fully-constructed `Model[Any]` instead of
`KnowledgeSource`, letting the caller handle model construction.

**Why rejected**: Breaks the `KnowledgeSource` abstraction. Range
extraction (`RangeExtractor`) and scope evaluation both need the source.
Passing a raw model would require duplicating the KB → Model translation
in every consumer and splitting the single `source` parameter into two.
