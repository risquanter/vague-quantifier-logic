# ADR-015: Value Type Boundaries — Safe Passage of Runtime Values

**Status:** Accepted  
**Date:** 2026-04-03  
**Tags:** type-safety, value-boundaries, runtime-types, dispatcher, literal-injection

---

## Context

- A heterogeneous sort system evaluates values of multiple declared types; a uniform carrier `Value(sort: TypeId, raw: Any)` is necessary because the library cannot enumerate consumer JVM types
- Every point where `Value.raw` crosses a boundary (into dispatcher lambdas or out to consumer result handlers) is an implicit cast site — unchecked by the compiler unless a disciplined pattern is enforced
- Two structurally distinct boundaries exist in the typed evaluation pipeline: **literal injection** (query-time string literals flowing into dispatcher arguments) and **result extraction** (evaluation output flowing into consumer domain types); each has different characteristics and requires a different mechanism
- Literal injection raw types are drawn from a finite, library-controlled set (integer, float, text) — a sealed type can provide exhaustive matching; result extraction raw types are open-ended consumer-defined types the library cannot enumerate
- A `TypeId` names a sort but carries no evidence of which JVM type the consumer chose to represent it; that mapping must be declared once, not assumed at each use site

---

## Decision

### 1. Injection Boundary — `LiteralValue` enum

Query-string literals (inline numerics and text) flow into dispatcher arguments as `LiteralValue`, a library-controlled enum type. The consumer pattern-matches exhaustively — no `asInstanceOf`.

```scala
// Library — enum, finite, library-controlled (ADR-006)
enum LiteralValue:
  case IntLiteral(value: Long)
  case FloatLiteral(value: Double)
  case TextLiteral(value: String)

// TypeCatalog — validators now produce the parsed value
literalValidators: Map[TypeId, String => Option[LiteralValue]]

// Example registration (consumer)
literalValidators = Map(
  TypeId("Loss") -> (s => s.toLongOption.map(IntLiteral(_))),
  TypeId("Prob") -> (s => s.toDoubleOption.map(FloatLiteral(_)))
)

// Dispatcher lambda — exhaustive match, no asInstanceOf
SymbolName("gt_prob") -> { case List(a, b) =>
  def toDouble(v: Value): Double = v.raw match
    case FloatLiteral(d) => d
    case IntLiteral(n)   => n.toDouble
  Right(toDouble(a) > toDouble(b))
}
```

### 2. Extraction Boundary — `TypeRepr[A]` + `Value.as[A]`

After evaluation, consumer domain types are projected from `EvaluationOutput[Value]` using a `TypeRepr[A]` given instance declared once by the consumer. The compiler verifies the mapping exists; the runtime sort check eliminates blind-cast failures.

```scala
// Library — mechanism
trait TypeRepr[A]:
  def typeId: TypeId

object Value:
  extension (v: Value)
    def as[A](using repr: TypeRepr[A]): Option[A] =
      if v.sort == repr.typeId then Some(v.raw.asInstanceOf[A]) else None

// Consumer declares mapping once
given TypeRepr[LeafId] with
  val typeId = TypeId("Asset")

// Consumer extracts at use site — compiler verifies TypeRepr exists
val satisfyingLeaves: Set[LeafId] =
  output.satisfyingElements.flatMap(_.as[LeafId])
```

### 3. Boundary Ownership Is Non-Negotiable

The injection boundary is owned by the library (`LiteralValue` is sealed, validators are library-registered at catalog construction). The extraction boundary is owned by the consumer (`TypeRepr` instances are declared in the consuming project). Neither boundary is optional — omitting either lands unchecked casts scattered at call sites.

## Code Smells

### ❌ Bare `asInstanceOf` at Dispatcher Call Sites

```scala
// BAD: origin of raw is implicit — breaks silently when a literal is passed
SymbolName("gt_prob") -> { case List(a, b) =>
  Right(a.raw.asInstanceOf[Double] > b.raw.asInstanceOf[Double])
}

// GOOD: sealed match — compiler forces exhaustiveness
SymbolName("gt_prob") -> { case List(a, b) =>
  def toDouble(v: Value) = v.raw match
    case FloatLiteral(d) => d
    case IntLiteral(n)   => n.toDouble
  Right(toDouble(a) > toDouble(b))
}
```

### ❌ Constructing `Value` Directly in a Function Lambda

```scala
// BAD: consumer builds Value with a native raw type;
// downstream comparison lambdas see Double here but LiteralValue from inline literals
// — two raw shapes for the same sort, requiring rawToDouble-style helpers
symLec -> { args =>
  val result: Double = computeLec(args)
  Right(Value(tProb, result))  // raw is Double — inconsistent with ConstRef path
}

// GOOD: return LiteralValue directly — framework wraps as Value(resultSort, literalResult)
symLec -> { args =>
  val result: Double = computeLec(args)
  Right(FloatLiteral(result))  // raw is FloatLiteral — consistent with ConstRef path
}

// ALSO GOOD: TypedFunctionImpl.of[A] makes the wrap declaration explicit
symLec -> TypedFunctionImpl.of[Double](
  impl = args => Right(computeLec(args)),
  wrap = FloatLiteral(_)
)
```

### ❌ Bare `asInstanceOf` at Result Extraction Sites

// GOOD: TypeRepr is the single declaration; Value.as checks sort at runtime
val leaves = output.satisfyingElements.flatMap(_.as[LeafId])
```

### ❌ `String => Boolean` Validators That Discard Parsed Values

```scala
// BAD: parses "10000000" to validate, then throws away the Long
literalValidators = Map(TypeId("Loss") -> (s => s.toLongOption.isDefined))
// Dispatcher must re-parse: v.raw.asInstanceOf[String].toLong

// GOOD: validator produces the parsed value once
literalValidators = Map(TypeId("Loss") -> (s => s.toLongOption.map(IntLiteral(_))))
// Dispatcher receives IntLiteral(10000000L) — no re-parsing
```

### ❌ Conflating the Two Boundaries

```scala
// BAD: using TypeRepr to handle literal values, or LiteralValue for domain results
// TypeRepr is for open-ended consumer types (LeafId, RiskId)
// LiteralValue is for the finite set of inline query literals
// Mixing them obscures which mechanism applies at which site

// GOOD: injection boundary → LiteralValue + sealed match
//       extraction boundary → TypeRepr[A] + Value.as[A]
```

## Cross-ADR Relationship

ADR-013 governs the extraction boundary in full detail and is the reference implementation for `TypeRepr[A]` and `Value.as[A]`. This ADR subsumes ADR-013 by placing it as one application of the general two-boundary principle. ADR-013 remains valid as the definitive source for extraction boundary usage and code examples; this ADR provides the framing that explains why two distinct mechanisms are required and why neither can substitute for the other.

ADR-001 governs compile-time sort correctness during query binding (`QueryBinder`). That boundary is upstream of both boundaries defined here — sort correctness at bind time is the precondition that makes the runtime sort check in `Value.as[A]` always succeed for well-formed queries.

```
[ADR-001]             [ADR-015 §1]                  [ADR-015 §2 / ADR-013]
ParsedQuery           LiteralValue injected          EvaluationOutput[Value]
  ──bind──>           into dispatcher args            ──.as[A]──> Set[LeafId]
  BoundQuery          (sealed, exhaustive match)      (TypeRepr, sort-checked)
```

## Alternatives Rejected

### `String => Option[Any]` for Literal Validators

- **What**: validators return `Option[Any]`; `ConstRef.raw: Any`; dispatcher uses `asInstanceOf`
- **Why rejected**: moves the cast from the re-parsing site to the dispatcher lambda, but does not eliminate it. Every dispatcher author must know and document the JVM type their validator produces. No compiler enforcement. Violates the spirit of the injection boundary principle defined in this ADR.

### `String => Boolean` Validators (Parsing Without Retention)

- **What**: validators validate the source text but return only `Boolean`; parsed value is discarded; `ConstRef` stores the source string; dispatcher re-parses on every evaluation
- **Why rejected**: semantically equivalent to no injection boundary at all. Re-parsing on every evaluation is redundant work. Dispatcher receives `String` raw for literals and native types for domain elements — two incompatible raw shapes for the same sort, with no compiler signal distinguishing them.
