# ADR-015: Symmetric Value Boundaries — Typeclass-Driven Injection and Extraction

**Status:** Proposed (under refactor; NOT binding until consistency review passes — see footer)
**Date:** 2026-05-01
**Tags:** type-safety, value-boundaries, runtime-types, dispatcher, typeclass, scala3
**Supersedes:** ADR-015 (2026-04-03 version, archived as `ADR-015.backup-2026-05-01.md`)

---

## Context

- A heterogeneous sort system evaluates values of multiple declared types; a uniform carrier `Value(sort: TypeId, raw: Any)` is necessary because the library cannot enumerate consumer JVM types.
- Every point where `Value.raw` crosses a boundary (parser-injected literal entering a dispatcher; evaluator output extracted by consumer code) is an implicit cast site — unchecked unless a disciplined pattern is enforced.
- Both boundaries — *injection* (literals from query text into dispatcher arguments) and *extraction* (evaluation output into consumer types) — face the same open-world constraint: the carrier JVM type for a given sort is the consumer's choice and is not enumerable by the library.
- A sealed-enum injection mechanism (`LiteralValue { IntLiteral | FloatLiteral | TextLiteral }`) cannot fit consumer carriers outside that closed set without forcing a re-resolution step (the source of T-002's named-constant defect: tree-supplied values and literal-injected values arrive in two distinct shapes for the same sort).
- The library cannot prevent a consumer from writing `asInstanceOf` at boundaries; what it can do is provide a single declaration site per (sort, carrier) pair and a parametric API that makes ad-hoc casts unnecessary.

---

## Decision

### 1. `LiteralParser[A]` typeclass — injection boundary

Query-string literals are parsed into the consumer's chosen carrier `A` via a typeclass instance. The library provides instances for primitive carriers; consumers provide instances for their own carriers.

```scala
// Library
trait LiteralParser[A]:
  def parse(s: String): Option[A]

object LiteralParser:
  given LiteralParser[Long]   with def parse(s: String) = s.toLongOption
  given LiteralParser[Double] with def parse(s: String) = s.toDoubleOption
  // String parser intentionally not given — consumers opt in per-sort

// TypeCatalog signature
literalValidators: Map[TypeId, String => Option[Any]]

// Consumer registers per sort
literalValidators = Map(
  TypeId("Loss")        -> summon[LiteralParser[Long]].parse,
  TypeId("Probability") -> summon[LiteralParser[Double]].parse,
  TypeId("Asset")       -> AssetParsers.fromToken    // consumer-defined String => Option[String]
)
```

### 2. `Extract[A]` typeclass — extraction boundary

Values are extracted from `Value` into the consumer's chosen carrier via the same typeclass shape. Library provides instances for primitive carriers (with numeric widening); consumers provide instances for their own carriers.

```scala
// Library
trait Extract[A]:
  def fromRaw(raw: Any): Either[String, A]

object Extract:
  given Extract[Long]   with /* widens Int -> Long */
  given Extract[Double] with /* widens Long, Int -> Double */
  given Extract[String] with /* exact match only */

extension (v: Value)
  def extract[A](using e: Extract[A]): Either[String, A] = e.fromRaw(v.raw)
```

### 3. Single declaration site per (sort, carrier)

Per-sort consumer registration appears exactly once at catalog construction (`literalValidators`) and exactly once per consumer-defined carrier as a `given Extract[A]`. Dispatcher arms call `args(i).extract[A]`; result projection calls `value.extract[A]`. No `asInstanceOf`, no per-call match arms.

```scala
// Dispatcher (consumer side)
case "leaf_descendant_of" =>
  for
    desc     <- args(0).extract[String]
    ancestor <- args(1).extract[String]
  yield descendantsByName.getOrElse(ancestor, Set.empty).contains(desc)
```

### 4. `QueryBinder` named-constant branch routes through `literalValidators`

A query name resolved as `catalog.constants.get(name)` is bound by passing `name` through the registered validator for its sort. `ConstRef.raw: Any` carries the validator's output (matching the carrier shape used everywhere else for that sort). Resolves T-002.

```scala
// QueryBinder, named-constant branch
catalog.constants.get(name) match
  case Some(actual) if actual == expected =>
    catalog.literalValidators.get(expected).flatMap(_(name)) match
      case Some(parsed) => Right((BoundTerm.ConstRef(name, expected, parsed), env))
      case None         => Left(BindError.UnparseableConstant(name, expected))
```

### 5. Boundary ownership is non-negotiable

The injection mechanism is owned by the library (`LiteralParser` trait, library-provided primitive instances). The extraction mechanism is owned by the library (`Extract` trait, library-provided primitive instances). Per-sort wiring (`literalValidators` map entries) and consumer-carrier instances (`given Extract[MyType]`) are owned by the consumer. Neither side can be omitted without re-introducing scattered casts.

## Code Smells

### ❌ Bare `asInstanceOf` at dispatcher call sites

```scala
// BAD: origin of raw is implicit; breaks silently when carrier shape changes
SymbolName("gt_prob") -> { case List(a, b) =>
  Right(a.raw.asInstanceOf[Double] > b.raw.asInstanceOf[Double])
}

// GOOD: typeclass-driven; missing Extract[A] is a compile error
SymbolName("gt_prob") -> { case List(a, b) =>
  for { x <- a.extract[Double]; y <- b.extract[Double] } yield x > y
}
```

### ❌ Per-sort hand-written extract helper

```scala
// BAD: same shape repeated per sort in every consumer
def extractString(args: List[Value], idx: Int, ctx: String): Either[String, String] =
  args.lift(idx).toRight(s"$ctx: missing").flatMap { v =>
    v.raw match { case s: String => Right(s); case _ => Left(s"$ctx: bad type") }
  }

// GOOD: single library extension; consumer writes nothing
args(idx).extract[String]
```

### ❌ Constructing `Value` directly in a function lambda

```scala
// BAD: consumer chooses an ad-hoc raw type; downstream extract instances may not match
symLec -> { args =>
  Right(Value(tProb, computeLec(args)))   // raw is Double — shape contract is implicit
}

// GOOD: framework wraps via the same typeclass machinery; carrier consistent across paths
symLec -> { args => Right(computeLec(args)) }   // returns Double; framework wraps
```

### ❌ `String => Boolean` validators that discard parsed values

```scala
// BAD: parses to validate, throws away the parsed value, dispatcher must re-parse
literalValidators = Map(TypeId("Loss") -> (s => s.toLongOption.isDefined))

// GOOD: validator IS the parser; dispatcher receives the parsed carrier
literalValidators = Map(TypeId("Loss") -> summon[LiteralParser[Long]].parse)
```

### ❌ Two carrier shapes for the same sort

```scala
// BAD: tree path stores raw String; literal path stores wrapped TextLiteral
//      dispatcher arm must match both — the T-002 defect shape

// GOOD: consumer's LiteralParser instance produces the same carrier the tree path stores
//      one Extract[A] arm covers everything
```

## Cross-ADR Relationship

ADR-001 governs compile-time sort correctness during query binding (`QueryBinder`). That boundary is upstream of both boundaries defined here — sort correctness at bind time is the precondition that makes the runtime carrier check inside `Extract[A]` always succeed for well-formed queries.

ADR-013 (`TypeRepr[A]` for consumer domain extraction) is subsumed by §2 of this ADR. `Extract[A]` plays the same role with a more general signature (`Either[String, A]` rather than `Option[A]`) and a single mechanism shared with the injection side. ADR-013 should be marked **superseded by ADR-015 (2026-05-01)** when this ADR is accepted.

ADR-006 (enum encoding for pure-data sums) applied to the prior `LiteralValue`. With `LiteralValue` removed, ADR-006 governance shifts to the ADTs that remain (e.g. `BoundTerm`, `Formula`).

## Alternatives Rejected

### Sealed `LiteralValue` enum on the injection side (the prior 2026-04-03 ADR-015 §1)
- **What:** validators return a sealed `LiteralValue` (Int/Float/Text); dispatchers exhaustively match.
- **Why rejected:** the sealed set fits only sorts whose carrier is one of three primitive shapes. For any other consumer carrier the validator must return `TextLiteral(sourceToken)` and the dispatcher re-resolves, producing two carrier shapes for the same sort (the T-002 defect). The asymmetry it created — sealed injection vs open extraction — was a design choice, not a forced one; both directions face the same open-world constraint and benefit from the same parametric mechanism.

### `String => Option[Any]` validators with bare `asInstanceOf` at dispatcher
- **What:** validators return `Option[Any]`; dispatcher casts.
- **Why rejected:** moves the cast site rather than eliminating it. No compile-time enforcement that the validator's output matches what the dispatcher expects.

### Match-typed `Value[S <: TypeId]` with compile-time carrier mapping
- **What:** `type CarrierOf[S] = S match { case LossType => Long; … }`, then `Value[S](sort: S, raw: CarrierOf[S])`.
- **Why rejected:** requires sorts to be singleton types known at compile time. The library's `TypeCatalog` is built at runtime from query text; sorts are values, not types. Workable only for closed-world sort sets, which contradicts the library's openness.

### Closed-world specialisation of `Value` to a fixed carrier set
- **What:** abandon `Value.raw: Any`; declare the library's supported carriers as a sealed enum.
- **Why rejected:** forfeits VQL's reusability for consumers with non-primitive carriers; collapses ADR-008's `KnowledgeBase[D]` generic ambition.
