# ADR-012: Error Channel Policy — `require` vs `Either`

**Status:** Accepted  
**Date:** 2026-03-26  
**Tags:** error-handling, Either, require, preconditions, typed-errors

---

## Context

- Two distinct failure modes exist: **programmer error** (broken invariants, impossible states) and **operational failure** (missing relation, bad input, I/O error)
- Scala's `require()` throws `IllegalArgumentException` — untyped, unchecked, invisible in the signature
- `Either[QueryError, A]` makes failure explicit in the return type — callers must handle it
- The OCaml-ported foundation layer (ADR-007 scope) uses exceptions for internal control flow by design; this is a known special case, not a contradiction — see §Relationship to ADR-007 below
- Constructor invariants (e.g. "arity ≥ 1", "name non-empty") are properties of the **type itself** — once constructed, the value is guaranteed valid
- Operational methods (e.g. "look up relation X in the KB") depend on **external state** the caller does not control

---

## Decision

### 1. `require()` — Construction-Time Invariants Only

Use `require()` for invariants that define what a value **is**. These are preconditions on the type's own construction — a violation is a programming error, not a recoverable condition.

```scala
// ✅ require: the type itself guarantees non-empty
opaque type RelationName = String
object RelationName:
  def apply(name: String): RelationName =
    require(name.nonEmpty, "Relation name must not be empty")
    name

// ✅ require: structural invariant of the data type
case class Relation(name: RelationName, arity: Int, ...):
  require(arity >= 1, "Relation must have at least arity 1")
```

### 2. `Either[QueryError, A]` — Operational Failures

Use `Either` for any method whose success depends on state external to its arguments. The caller cannot guarantee the precondition holds without inspecting the state first — that inspection-then-act pattern is the duplication `Either` eliminates.

```scala
// ✅ Either: success depends on KB state (does the relation exist?)
def getDomain(name: RelationName, pos: Int): Either[QueryError, Set[D]]

// ✅ Either: success depends on schema (arity match, position bounds)
def addFact(name: RelationName, tuple: RelationTuple[D]): Either[QueryError, KnowledgeBase[D]]
```

### 3. `require()` as Belt-and-Suspenders Behind `Either` Boundaries

A `require()` MAY remain as a redundant safety net inside a method that is already validated by an `Either`-returning caller. The `require` must never be the **primary or only** error handling — it exists to catch invariant violations that "should be impossible".

```scala
// ✅ Belt-and-suspenders: KB.query validates relation exists (Either),
//    then delegates to matches() which has a redundant require
case class RelationTuple[D](values: List[D]):
  def matches(pattern: List[Option[D]]): Boolean =
    require(pattern.length == values.length, "Pattern length must match tuple arity")
    // ...
```

### 4. Decision Criteria — Quick Reference

| Question | `require()` | `Either` |
|---|---|---|
| Can the caller guarantee the precondition at the call site? | Yes | No |
| Does success depend on external/mutable state? | No | Yes |
| Is failure a programming error or a runtime condition? | Programming error | Runtime condition |
| Would migrating to Either force every constructor call to unwrap? | Yes → keep `require` | No |
| Is the check inside OCaml-ported code (ADR-007 scope)? | Use `throw` (C2, C12) | N/A |

## Relationship to ADR-007

ADR-007 mandates exception-based control flow in the OCaml-ported foundation layer (parser combinators, lexer, FOL semantics, printer). That mandate is **not in conflict** with this ADR:

- ADR-007 applies to a **closed set of files** listed in its scope table (Tiers 1–3). Those files use `throw` for internal backtracking and `failwith`-style signalling. Public boundaries in those files already convert to `Either` (e.g., `FOLParser.parse`, `RangeExtractor.extractRange`).
- ADR-012 applies to **all other code** — the `fol/` application layer, bridge, datastore, quantifier, query, and sampling packages.
- Where OCaml-style internals inject closures into the evaluation engine (e.g., augmenter functions stored in `Interpretation[D]`), their throws are caught at the existing `try/catch` boundary in `VagueSemantics.holds`/`.evaluate`. Migrating those closures would require changing `Interpretation`'s function signature type, which falls under ADR-007's scope — and is therefore out of scope for ADR-012.

**Summary:** ADR-007 governs the OCaml core; ADR-012 governs everything else. The boundary between them is the `try/catch` at each public API method.

## Code Smells

### ❌ `require()` as the Only Guard on State-Dependent Operations

```scala
// BAD: success depends on KB state, but require throws untyped
def getDomain(name: RelationName, pos: Int): Set[D] =
  val rel = schema.get(name).getOrElse(
    throw new IllegalArgumentException(s"Unknown relation: ${name.value}"))
  // ...

// GOOD: state-dependent failure is explicit in the type
def getDomain(name: RelationName, pos: Int): Either[QueryError, Set[D]] =
  schema.get(name) match
    case None => Left(QueryError.RelationNotFoundError(name, schema.keySet))
    case Some(rel) => // ...
```

### ❌ `Either` for Trivial Construction Invariants

```scala
// BAD: forces every call site to unwrap for a trivial invariant
def apply(name: String): Either[QueryError, RelationName] =
  if name.isEmpty then Left(QueryError.ValidationError("empty", "name"))
  else Right(name)

// GOOD: require — caller controls the input, failure is programming error
def apply(name: String): RelationName =
  require(name.nonEmpty, "Relation name must not be empty")
  name
```

### ❌ Throwing in a Non-OCaml Method That Could Return Either

```scala
// BAD: private helper converts Left to throw, public method catches it back
private def toResolved(...): ResolvedQuery[D] =
  val range = RangeExtractor.extractRange(...) match
    case Right(elems) => elems
    case Left(error) => throw error.toThrowable  // throw-to-catch trampoline
// ...
try Right(toResolved(...).evaluate())
catch case e: QueryException => Left(e.error)

// GOOD: compose Either directly
private def toResolved(...): Either[QueryError, ResolvedQuery[D]] =
  RangeExtractor.extractRange(...).map { elems => ... }
```

## Implementation

| Location | Pattern |
|----------|---------|
| `RelationName.apply` | `require` — construction invariant |
| `RelationTuple(values)` | `require` — construction invariant |
| `Relation(name, arity, props)` | `require` — construction invariant |
| `VagueQuantifier` case classes | `require` — construction invariant |
| `Quantifier.mk*` smart constructors | `require` — construction invariant |
| `Quantifier.accepts` | `require` — belt-and-suspenders (values computed internally) |
| `Domain(elements)` | `require` — construction invariant (OCaml core, ADR-007) |
| `SamplingParams`, `HDRSampler`, `NormalApprox` | `require` — construction invariant |
| `KnowledgeBase.getDomain/query/contains/count/addFact/addFacts/addRelation` | `Either` — state-dependent (TD-001) |
| `KnowledgeSource` trait methods | `Either` — state-dependent (TD-001) |
| `VagueSemantics.toResolved` | `Either` — eliminates throw-to-catch trampoline |
| `RelationTuple.matches` | `require` — belt-and-suspenders behind `Either` boundary |
